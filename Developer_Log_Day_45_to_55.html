Executive Summary: The Condition Then vs. Now

Where We Were (Day 45 Condition)

At Day 45, Project Pandora had a fragmented architecture. The platform was struggling with several critical issues:

  • Monolithic Editors: We had legacy monolith files (like editor_template1.py and isolated templates) running outside the core Flask app. Synchronization was broken.
  • NLP Brittleness: The "Extractor Engine" produced poor quality summaries, often choking on symbols or generating generic categorizations like "general" for every article.
  • Deployment Blockers: Features like the Git Image Pusher relied on local system commands and user identities, completely failing in the Render.com cloud environment.
  • Timeout Failures: Our Batch Extractor API used synchronous JSON requests. Generating AI summaries for 5 URLs routinely exceeded Render's 30-second timeout, resulting in 502 Bad Gateway errors.
  • API Rate Limiting: Gemini API keys were hardcoded and singular. The app frequently crashed due to rate limit exhaustion during batch processing.
Where We Are Now (Day 55 Condition)

Over the last 10 days, we've executed a massive architectural shift to a resilient, modular, and cloud-native system:

  • Unified Flask Blueprints: All editors (Day 8, Day 9, Template 1, Day 11, Day 12, Day 15) are now fully encapsulated as Blueprints routed perfectly through app.py.
  • Robust NLP Pipeline: Event categorization now uses weighted semantic matching and rule-based fallback, ditching "general" for specific tags like "Geopolitics". The summarization strictly caps at 30 words.
  • Cloud-Native Workflows: GitHub image pushing is authenticated via GITHUB_TOKEN from .env, totally bypassing local Git identity constraints.
  • SSE Streaming Architecture: The Batch Extractor was rebuilt using Server-Sent Events (SSE) with heartbeat keepalives. 30-second timeouts are a thing of the past.
  • Dynamic Key Rotation: Built a thread-safe, N-key rotation system for the Gemini API to mathematically eliminate quota exhaustion.

Architecture 1. Batch Extractor SSE & API Key Rotation Upgrade

The batch_extractor.py was previously our biggest point of failure in production.

The 502 Bad Gateway Crisis

The Issue: When sending 5 URLs to the Batch Extractor, the backend would iterate through them synchronously. Calling the Gemini API 5 times took roughly 35-45 seconds. Render enforces a strict 30-second idle timeout. As a result, the connection would drop, the frontend would receive an HTML error page instead of JSON (causing the "JSON parsing error: Unexpected token '<'"), and the entire batch was lost.

Secondary Issue: Hitting Gemini in rapid succession with a single free-tier API key led to instantaneous 429 Too Many Requests errors.

The SSE Streaming Solution

We completely refactored the extraction endpoint from a standard JSON POST to an SSE (Server-Sent Events) stream. Instead of waiting for all 5 to finish, we yield results instantly as they complete. Crucially, we implemented a keepalive heartbeat.

@batch_extractor_bp.route("/api/batch-stream")
def api_batch_stream():
    def generate():
        yield ': keepalive\n\n' # Immediate heartbeat
        
        result_q = queue.Queue()
        # Threaded worker implementation...
        
        while next_to_emit < total:
            try:
                idx, story = result_q.get(timeout=5)
                # Ensure strict URL-to-story ordering
                buffer[idx] = story
                while next_to_emit in buffer:
                    s = buffer.pop(next_to_emit)
                    yield f'event: result\ndata: {json.dumps(s)}\n\n'
                    next_to_emit += 1
            except queue.Empty:
                # Keepalive to prevent Render 30s timeout
                yield ': keepalive\n\n'
                
        yield 'event: done\ndata: complete\n\n'

Dynamic N-Key Rotation

To fix the rate limit, we implemented a regex-based environment scanner that loads an infinite number of API keys (GEMINI_API_KEY_1, GEMINI_API_KEY_2, etc.) and rotates them per request using a threading.Lock().

def _discover_api_keys() -> list[str]:
    import re
    found = {}
    for name, val in os.environ.items():
        m = re.fullmatch(r'GEMINI_API_KEY_(\d+)', name)
        if m and val and val.strip():
            found[int(m.group(1))] = val.strip()
    return [found[k] for k in sorted(found)]

def _next_key() -> str:

Feature 2. NLP Pipeline & Event Categorization Polish

We needed to move beyond the generic metadata generated by our earliest iterations.

Previous Condition

The system was using a generic classify_event that defaulted almost everything to "General" or "Latest News". The summarization pipeline had no length enforcement, leading to UI overflow in the newsletter templates.

Strict Prompting and Length Enforcement

We implemented a two-pass quality gate. The first pass summarizes, the second pass enforces a hard 30-word cutoff algorithmically to ensure design consistency in the templates.

def summarize_short(content: str) -> str:
    prompt = (
        "Summarize the following text in 25 words or fewer. "
        "Be specific and preserve key facts. "
        "Reply with only the summary, nothing else.\n\n"
        f"Text: {content}"
    )
    result = _call_gemini(prompt)
    if not result: return content
    words = result.split()
    return " ".join(words[:30]) if len(words) > 30 else result

Bugfix 4. Render.com Cloud Migration: GitHub Token Auth

One of the hardest blockers during this period was migrating our Git Image Pusher tool from a local developer environment to the Render hosting environment.

The Condition

The Git Image Pusher relied on os.system("git commit -m 'add image'"). Locally, this worked because Git was configured with the user's name and email globally. On Render, the container is ephemeral and lacks Git configuration, causing every push to crash with "Please tell me who you are". Additionally, SSH keys weren't configured.

Implementing GITHUB_TOKEN Workarounds

We abandoned SSH and global git config. We modified the backend to read a GITHUB_TOKEN from the .env file. We then dynamically injected the identity and the auth token into the shell commands.

# Render environment bypass logic for Git
import subprocess
import os

token = os.getenv("GITHUB_TOKEN")
repo_url = f"https://oauth2:{token}@github.com/Manas236/Render_Hosting_2.git"

# Forcing identity on the fly
subprocess.run(["git", "-c", "user.name=PandoraBot", "-c", "user.email=bot@pandora.local", "commit", "-m", "Auto-uploaded image via dashboard"], check=True)

# Pushing securely
subprocess.run(["git", "push", repo_url, "main"], check=True)
