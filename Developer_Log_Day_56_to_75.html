Executive Summary: Day 55 Condition vs. Day 75 Condition

Where We Were (Day 55 Condition)

At the close of Day 55, Project Pandora had reached a stable first-generation architecture. The SSE batch extractor, API key rotation, and a fleet of six editors (Day 6 Temp, Day 8, Day 9, Day 11, Day 12, Day 15) were fully integrated. However, several fragility vectors remained:

  • SDK Coupling: Both batch_extractor.py and extractor.py depended on the google-genai Python package. Any breaking change to this SDK — or an environment where it failed to import — would crash the entire application on startup with a hard sys.exit().
  • Single-Point Gold Scraper: The Mumbai gold price scraper had no fallback. If goodreturns.in returned a status other than 200, or changed its HTML structure, the markets ticker would silently fail with no data.
  • USD/INR via Equity API: The exchange rate was fetched using fetch_two("INR=X") — the same Yahoo Finance function designed for equity indices — which returned inconsistent granularity and lacked a fallback.
  • Day 15 Compact Story Selector Fragility: The _find_compact_stories() function relied on sequential element-counting across the entire document DOM. Adding any new element of the same CSS type elsewhere in the template would silently map to the wrong story.
  • No Variant Templates: The library had one version of each day's design. There was no mechanism for issuing an alternate layout for a day where the editorial brief changed mid-production.
  • No Diagnostic Tool: There was no standalone script to validate that all blueprint modules could be imported successfully — a startup crash on Render required reading cloud logs to identify the failing module.
Where We Are Now (Day 75 Condition)

Over the last 20 development days, every one of these fragility vectors has been resolved, and the platform has expanded significantly:

  • SDK-Free Gemini Calls: Both batch_extractor.py and extractor.py now call the Gemini REST API directly via requests.post(). The google-genai import has been eliminated entirely, reducing the dependency surface and removing the hard startup crash.
  • Three-Tier Gold Fallback: Mumbai 24K gold scraping now has a fully documented three-tier fallback chain: goodreturns.in Mumbai → goodreturns.in national → Yahoo Finance COMEX conversion. The scraper will never silently fail as long as any one of these three sources is reachable.
  • Dedicated USD/INR Resolver: A new fetch_usd_inr() function uses the Frankfurter API (European Central Bank rates) as its primary source, with Yahoo Finance as fallback. This is more semantically correct than using an equity ticker for a currency rate.
  • Wrapper-Scoped Story Selectors: The Day 15 compact story finder now locates articles by finding their wrapping <a> element first, then searches within that scope. Adding new elements to the document no longer risks cross-contamination between stories.
  • Two New Variant Templates: Day12(2).html and Day9(2).html have been introduced as production-grade alternate layouts, each with their own full-stack editor implementation. The newsletter fleet now includes a total of nine template variants.
  • Day 17 Newsletter: The newest edition template has been added to the platform and wired into the full pipeline — editor backend, editor frontend, dashboard card, and batch extractor forwarding.
  • Universal Date Format: All editors now generate dates in Month DD, YYYY format (e.g., "May 22, 2025") instead of the previous DD/MM/YYYY, with backward-compatible parsing to avoid breaking existing content.
  • Image Pusher Validated in Production: Five consecutive live image uploads (commits 57–62) proved that the GitHub Token auto-push pipeline established in Day 45–55 is functioning correctly in the Render.com production environment.
  • Diagnostic Import Harness: test_imports.py provides a standalone script that sequentially imports all modules and prints a clear success/failure status for each, making deployment debugging dramatically faster.

Migration 1. Gemini SDK → Direct REST API: Eliminating google-genai

The single most architecturally significant change in this period was removing the dependency on the google-genai Python package from both batch_extractor.py and extractor.py. This was not a minor refactor — it changed how every AI call in the platform is executed.

The Fragility of SDK-Gated Startup

Both files had an unconditional hard-exit guard at the top of the module:

try:
    from google import genai
except ImportError:
    sys.exit("Missing dependency: pip install google-genai")

This meant that if the google-genai package was absent, outdated, or incompatible with the Python version on Render, the entire Flask application would refuse to start. The Blueprint registration in app.py uses importlib.util.exec_module() to load these files — so a failed import in either module propagated as an unhandled exception that took down the whole app, not just the affected blueprint.

Furthermore, the genai.Client object was being constructed inside the core API-calling functions (_call_gemini, summarize, categorize), meaning one bad API key could cause the SDK to raise during client construction rather than during the network call, producing confusing error messages.

The New Architecture: Raw HTTP via requests

The fix was to bypass the SDK entirely and call the Gemini REST endpoint directly using the requests library — which was already a dependency. The requests import was already soft-guarded with a try/except, so no new hard dependencies were introduced.

Before (SDK approach — both files)

from google import genai                        # Hard import, crashes on missing

client = genai.Client(api_key=api_key)          # Constructed per-call
for model in GEMINI_MODELS:
    response = client.models.generate_content(
        model=model, contents=prompt
    )
    if response and response.text:
        return response.text.strip()

After (REST approach — both files)

# No google-genai import needed at all.

for model in GEMINI_MODELS:
    try:
        url = (
            f"https://generativelanguage.googleapis.com"
            f"/v1beta/models/{model}:generateContent?key={api_key}"
        )
        headers = {'Content-Type': 'application/json'}
        data = {"contents": [{"parts": [{"text": prompt}]}]}

        resp = requests.post(url, headers=headers, json=data, timeout=30)
        if resp.ok:
            res_json = resp.json()
            candidates = res_json.get("candidates", [])
            if candidates and candidates[0].get("content", {}).get("parts"):
                return candidates[0]["content"]["parts"][0]["text"].strip()
        elif resp.status_code == 429:
            logger.warning(f"[!] Rate limited {model} key {masked}")
        else:
            logger.error(f"[!] {model} failed: HTTP {resp.status_code}")
            break
    except Exception as e:
        logger.error(f"[!] {model} request failed: {str(e)[:100]}")
        break
What This Achieves
  • Zero hard dependencies added: requests was already in use. The only removal is google-genai.
  • Explicit HTTP status handling: 429 rate limits are caught and logged with a proper warning. Non-200 statuses are logged with the HTTP code. The SDK previously wrapped these in SDK-specific exception types that were harder to pattern-match.
  • Startup safety: If the REST call fails, it logs an error and returns None — the app continues running. There is no sys.exit() anywhere in the call path.
  • Identical behaviour: The model rotation, key cycling, and retry logic are fully preserved. The change is purely at the transport layer.
Side Effect: Unicode Arrow Fix

As part of this refactor, two log lines in batch_extractor.py that used the Unicode arrow → were changed to the ASCII equivalent ->. The Unicode character caused terminal encoding errors on Windows consoles running cp1252 encoding, which would abort the log output mid-stream and scramble subsequent log lines. The fix is backward-compatible because no log parser depends on the specific character.

Editor Backend: BeautifulSoup4 Selector Strategy

The editor(for Day12(2).html).py backend follows the same BeautifulSoup4 pattern established by all previous editors. Its selector logic is designed for a document that uses <a> tags as article wrappers — a structure common to email-safe HTML where interactive areas must wrap entire table rows.

def _find_articles(soup):
    """
    Day12(2) has 5 articles, each wrapped in <a> with:
       style="text-decoration:none; display:block"
    Footer social links share the same anchor style but live inside
    td[background-color:#0a0a0a] — we filter those out.
    """
    articles = []
    for a_tag in soup.find_all("a"):
        style = a_tag.get("style", "")
        if "text-decoration:none" in style and "display:block" in style:
            parent_td = a_tag.find_parent("td")
            if parent_td:
                parent_style = parent_td.get("style", "")
                if "background-color:#0a0a0a" in parent_style:
                    continue          # skip footer links
            articles.append(a_tag)
    return articles
Why the Footer Filter Matters

Without the background-color:#0a0a0a filter, the footer social links (Instagram, Twitter, website) — which also use display:block; text-decoration:none anchors — would be appended to the articles list as phantom articles 6, 7, and 8. This would cause the editor to render three extra story panels that map to footer social icons, and any save operation would corrupt the footer HTML. The parent <td> background colour check is the only reliable discriminator between article anchors and footer anchors in this template's structure.

Market Section Selectors

The markets ticker strip is identified by the containing <td> with background-color:#0f172a. Within it, the four instrument cells are each a <td valign="top" width="25%">. Each cell contains three <div> elements in order: label, value, change percentage.

def _find_market_data(soup):
    dark_td = None
    for td in soup.find_all("td"):
        if "background-color:#0f172a" in td.get("style", ""):
            dark_td = td
            break

    ticker_cells = [
        td for td in dark_td.find_all("td")
        if td.get("valign") == "top" and td.get("width") == "25%"
    ]

    for cell in ticker_cells:
        divs = cell.find_all("div")
        if len(divs) >= 3:
            market = {
                "label":    divs[0].get_text().strip(),
                "value":    divs[1].get_text().strip(),
                "change":   divs[2].get_text().strip(),
                "positive": "color:#10b981" in divs[2].get("style", ""),
            }
            markets.append(market)

Refactor 3. Day 12(2) HTML Template: CSS & Mobile Overhaul

Before the editor integration, the Day12(2).html base template itself underwent structural CSS improvements that were committed as part of the same changeset.

BOM Character Removal

The file previously opened with a BOM (Byte Order Mark, EF BB BF), making the DOCTYPE declaration effectively invisible to some parsers. A BOM before <!DOCTYPE html> causes Internet Explorer and some email clients to enter quirks mode. It was removed, leaving a clean ASCII start of file.

Mobile CSS Rewrite

The previous media query used a standalone mobile-only breakpoint:

@media only screen and (max-width: 754px) {
    body { overflow-x: hidden !important; margin: 0 !important; }
    table.nb-wrapper { zoom: calc(100vw / 754) !important; }
}

The new version adds a landscape orientation breakpoint, which addresses the case where a phone is held horizontally — the viewport width exceeds 754px but the height is very small, causing the template to overflow vertically:

body { margin:0; padding:0; background-color:#e0e0e0;
       -webkit-text-size-adjust:100%; -ms-text-size-adjust:100%; }
table { border-collapse:collapse; }

/* Mobile + landscape — zoom-scale the whole shell to fit viewport */
@media only screen and (max-width: 754px),
       only screen and (orientation: landscape) and (max-height: 500px) {
    body { overflow-x:hidden !important; margin:0 !important; }
    table.nb-wrapper { zoom: calc(100vw / 754) !important; }
}

The body baseline styles are also moved into the stylesheet (from inline style attributes on the <body> tag) to reduce redundancy.

Box Shadow on the Wrapper Table

The main nb-wrapper table received a subtle drop shadow:

box-shadow: 0 4px 20px rgba(0,0,0,0.15);

This creates visual separation between the newsletter body and the grey page background when viewed in a browser preview — relevant for the editor's live preview iframe. It has no effect in email clients, which strip box-shadow.

Refactor 4. Day 15 Editor: Compact Story Selector Rewrite

The _find_compact_stories() function inside editor(for Day15.html).py was rebuilt from scratch. The old implementation was a textbook example of an order-dependent, document-global selector strategy — the kind that works perfectly until the template is edited once.

The Old Strategy and Its Fatal Flaw

The previous function worked by finding all elements of a given style across the entire document, then taking a positional slice:

# Old approach — DO NOT USE
compact_images    = [img for img that match style globally]
compact_headlines = [td  for td  that match font-size:21px globally]
compact_cats      = all_cat_spans[2:4]   # assumes exactly 2 twin stories before
compact_summaries = all_summaries[2:4]   # assumes exactly 4 summaries total
compact_links     = all_read_links[2:4]  # assumes exactly 2 other "Read →" links

This approach had three failure modes: (1) If a story was edited to add a new <td> with font-size:21px anywhere in the DOM, the index slice would shift by one, mapping the editor's "Compact Story 1" panel to the wrong story element. (2) If any article had no image, the image list would be shorter than the headline list, causing index-out-of-bounds silent mismatches. (3) It was impossible to add a third compact story to a future template variant without rewriting the entire function.

The New Wrapper-Scoped Strategy

Each compact story in Day 15 is wrapped in an <a> tag with display:block; text-decoration:none; color:inherit. The new function finds these wrappers first, then searches for each field within its own wrapper's subtree:

def _find_compact_stories(soup):
    compact_wrappers = []
    for a in soup.find_all("a"):
        style = a.get("style", "")
        if ("display:block" in style
                and "text-decoration:none" in style
                and "color:inherit" in style):
            for img in a.find_all("img"):
                if "max-width:240px" in img.get("style", ""):
                    compact_wrappers.append(a)
                    break

    for i, wrapper in enumerate(compact_wrappers[:2]):
        story = {"index": i, "link": wrapper}

        for img in wrapper.find_all("img"):
            if "max-width:240px" in img.get("style", ""):
                story["image"] = img
                break

        for span in wrapper.find_all("span"):
            if ("background:#8b2a1f" in span.get("style", "")
                    and "padding:4px 9px" in span.get("style", "")):
                story["category"] = span
                break

        for td in wrapper.find_all("td"):
            if ("font-size:21px" in td.get("style", "")
                    and "font-weight:700" in td.get("style", "")):
                story["headline"] = td
                break

        for td in wrapper.find_all("td"):
            if ("font-size:14px" in td.get("style", "")
                    and "line-height:1.55" in td.get("style", "")
                    and "color:#3a3a3a" in td.get("style", "")):
                story["summary"] = td
                break

        stories.append(story)
Why This Is Correct

Each field is now found by searching inside its own story's wrapper. Adding a new article above or below the compact section has zero effect on the selector — the wrappers are identified by a combination of three style properties that only appear on compact story anchors. Story count is naturally capped to two by the [:2] slice on the wrappers list. The link reference now points to the wrapper anchor itself, making the "open link" field in the editor directly usable rather than requiring a secondary selector.

Bugfix 5. Day 15 HTML Template: Compact Article Clickability

The Day 15 template's compact stories (the two image-and-text horizontal cards in the lower section) had a structural problem: the articles were not fully clickable. Only the "Read →" label text was linked — the image and the headline were visually part of the article but not tappable in email clients.

The Problem

Each compact story had an <a href="#">Read →</a> tag at the bottom of the text column. The image, category label, headline, and summary were all plain text and images with no enclosing anchor. On mobile email clients where the entire card should be a tap target, this meant users had to aim precisely at the two words "Read →" — a terrible touch experience.

The Fix: Wrapping Anchor + Span Conversion

Each compact story's outer <td> padding cell was wrapped in a block-level <a>:

<!-- Before -->
<td style="background:#faf7f1; padding:44px 56px 24px 56px;">
    <table ...>...article content...</table>
</td>

<!-- After -->
<td style="background:#faf7f1; padding:44px 56px 24px 56px;">
    <a href="#" target="_blank"
       style="display:block; text-decoration:none; color:inherit;">
        <table ...>...article content...</table>
    </a>
</td>

However, the existing <a href="#">Read →</a> inside the article content would then become a nested anchor — which is invalid HTML and causes unpredictable rendering in email clients. It was converted to a <span> with identical styling:

<!-- Before (nested <a> problem) -->
<a href="#" style="font-family:Arial; font-size:10px; letter-spacing:2.5px;
                   color:#1a1a1a; text-transform:uppercase; font-weight:700;
                   border-bottom:1px solid #1a1a1a; padding-bottom:3px;">
    Read &nbsp;→
</a>

<!-- After (valid structure) -->
<span style="font-family:Arial; font-size:10px; letter-spacing:2.5px;
             color:#1a1a1a; text-transform:uppercase; font-weight:700;
             border-bottom:1px solid #1a1a1a; padding-bottom:3px;">
    Read &nbsp;→
</span>

The two compact stories were both fixed with this pattern. The visual result is identical — the "Read →" affordance appears the same — but now the entire card area is the link target.

Image Cell: object-fit Fix

The image cells in the compact stories also had a height consistency issue — when the adjacent text column was taller than the image's natural height, the image column would compress. The fix adds object-fit:cover; min-height:200px:

<!-- Before -->
<td valign="top" width="38%" style="width:38%; padding-right:24px;">
    <img style="width:100%; max-width:240px; height:auto; display:block;">
</td>

<!-- After -->
<td width="38%" style="width:38%; padding-right:24px; overflow:hidden;">
    <img style="width:100%; max-width:240px; height:100%;
                min-height:200px; object-fit:cover; display:block;">
</td>

Refactor 6. Universal Date Format Standardization

Prior to this period, three editors (Day 9, Day 12, and Day 15) generated tomorrow's date in DD/MM/YYYY format. The new Day 12(2) and Day 9(2) editors were designed from the start to use the long-form Month DD, YYYY format. To unify the platform, the older editors were migrated.

Editor Old Format New Format
editor(for Day9.html).py dt.strftime("%d/%m/%Y") → "22/05/2025" dt.strftime("%B %d, %Y").replace(" 0", " ") → "May 22, 2025"
editor(for Day15.html).py dt.strftime("%d/%m/%Y") → "22/05/2025" dt.strftime("%B %d, %Y").replace(" 0", " ") → "May 22, 2025"
editor(for Day12(2).html).py N/A (new file) dt.strftime("%B %d, %Y").replace(" 0", " ") → "May 22, 2025"
editor(for Day9(2).html).py N/A (new file) dt.strftime("%B %d, %Y").replace(" 0", " ") → "May 22, 2025"
The .replace(" 0", " ") Detail

strftime("%B %d, %Y") on Python produces zero-padded day numbers: "May 05, 2025". The newsletter design calls for "May 5, 2025" without the leading zero. Since %-d (the non-padded day format) is a POSIX-only directive that doesn't work on Windows, the portable fix is to replace any occurrence of a space followed by zero (" 0") with a plain space. This correctly strips "May 05" → "May 5" without affecting the year or any other part of the string.

Backward-Compatible Date Rendering in Day 9 Editor

The Day 9 template's date is rendered inside the header as NEWSBAND · DD · MM · YYYY — a formatted string where the date components are split by middots. The old renderer expected a slash-delimited string:

# Old renderer — expected "22/05/2025"
parts = date_val.split("/")
if len(parts) == 3:
    _set_text(td, f"NEWSBAND · {parts[0]} · {parts[1]} · {parts[2]}")

The new renderer parses the long-form string first, with a fallback to the old slash format so that any existing saved HTML files are still rendered correctly when reopened:

# New renderer — accepts "May 22, 2025" or legacy "22/05/2025"
try:
    from datetime import datetime as _dt
    parsed = _dt.strptime(date_val, "%B %d, %Y")
    _set_text(
        td,
        f"NEWSBAND · {parsed.day:02d} · {parsed.month:02d} · {parsed.year}",
    )
except ValueError:
    parts = date_val.split("/")
    if len(parts) == 3:
        _set_text(td, f"NEWSBAND · {parts[0]} · {parts[1]} · {parts[2]}")

Architecture 7. Markets Data Resilience: Multi-Tier Fallback Chains

The market data fetching system — responsible for populating the live financial ticker in the Day 12(2), Day 15, and now Day 9(2) templates — received its most significant hardening upgrade to date. Two critical sources (Mumbai Gold and USD/INR) were each given a documented multi-source fallback chain.

Problem: Single-Point Scraper Failure

The original fetch_mumbai_gold() function in editor(for Day15.html).py had a single source: goodreturns.in/gold-rates/mumbai.html. It made one request and returned the result if successful. There was no fallback — if goodreturns.in returned a non-200 status, the gold column in the ticker would be empty with no explanation in the logs.

Similarly, USD/INR was fetched via fetch_two("INR=X") — the same generic Yahoo Finance equity-chart function used for Sensex and Nifty. The INR=X ticker returns closing prices from the equity market session, not a real-time forex rate. On market holidays, it could return stale two-day-old data without any indication.

Solution A: Three-Tier Gold Scraping

The new fetch_mumbai_gold() function — now present in both editor(for Day15.html).py and editor(for Day12(2).html).py — implements a three-tier fallback with explicit debug logging at each stage:

    # Tier 1: Mumbai scrape
    try:

                        headers=headers, timeout=10)
        if resp.status_code == 200:
            # Parse first table, row where cols[0] == '10' (for 10g)
            ...
            if today_val > 0 and yday_val > 0:

    except Exception as e:
        print(f"DEBUG [Gold]: Mumbai failed ({e}), trying fallback...")

    try:
        resp2 = _req.get("https://www.goodreturns.in/gold-rates/", ...)
        ...  # same parse logic, same validation guards

    # Tier 3: COMEX via Yahoo Finance (GC=F), converted to INR
    # Formula: (gold_usd_per_oz / 31.1035g) * 10g * usd_inr_rate
    try:
        gold_resp = requests.get("https://query2.finance.yahoo.com/..."
                                 "GC=F?interval=1d&range=5d", ...)

                                 "INR=X?interval=1d&range=5d", ...)
        usd_inr   = usd_closes[-1]
        today_inr = (gold_closes[-1] / 31.1035) * 10 * usd_inr
        prev_inr  = (gold_closes[-2] / 31.1035) * 10 * usd_inr
        return round(today_inr), round(prev_inr)
    except Exception as e:
        print(f"DEBUG [Gold]: COMEX fallback failed ({e})")

    return None, None
The Conversion Math

COMEX gold (Yahoo GC=F) is quoted in USD per troy ounce. One troy ounce = 31.1035 grams. The newsletter template displays the 10-gram price in INR, which is the standard Indian retail reference unit. The formula (usd_per_oz / 31.1035) × 10 × usd_inr correctly converts to INR per 10 grams. The result is rounded to the nearest integer because gold is typically quoted in whole rupees at Indian retail.

Solution B: Dedicated USD/INR Resolver

The USD/INR fetch was extracted from the generic fetch_two() equity-chart function into a purpose-built fetch_usd_inr() with a two-tier fallback:

def fetch_usd_inr():
    """
    Tier 1: Frankfurter API (api.frankfurter.app) — ECB daily rates, free, no auth.
    Tier 2: Yahoo Finance INR=X — forex ticker, falls back to market close.
    """
    import requests
    from datetime import datetime, timedelta

    # Tier 1: Frankfurter (European Central Bank rates)
    try:
        resp = requests.get(
            "https://api.frankfurter.app/latest?from=USD&to=INR",
            timeout=10
        )
        if resp.status_code == 200:
            data = resp.json()
            today_rate = float(data['rates']['INR'])
            today_date = data['date']                      # "YYYY-MM-DD"
            prev_dt = datetime.strptime(today_date, "%Y-%m-%d") - timedelta(days=1)
            resp_prev = requests.get(
                f"https://api.frankfurter.app/{prev_dt.strftime('%Y-%m-%d')}"
                f"?from=USD&to=INR", timeout=10
            )
            if resp_prev.status_code == 200:
                prev_rate = float(resp_prev.json()['rates']['INR'])
                return today_rate, prev_rate
    except Exception as e:
        print(f"DEBUG [USD/INR]: Frankfurter failed ({e}), trying Yahoo...")

    # Tier 2: Yahoo Finance INR=X
    try:
        closes = resp.json()['chart']['result'][0]['indicators']['quote'][0]['close']
        closes = [c for c in closes if c is not None]
        if len(closes) >= 2:
            return float(closes[-1]), float(closes[-2])
    except Exception:
        pass

    return None, None

The api_markets_fetch() call in both Day 12(2) and Day 15 editors was updated to use fetch_usd_inr() instead of fetch_two("INR=X"):

# Before
u_now, u_prev = fetch_two("INR=X")

# After
u_now, u_prev = fetch_usd_inr()
Why Frankfurter Over Yahoo for Forex

The Frankfurter API (api.frankfurter.app) is a free, publicly accessible wrapper around the European Central Bank's official daily exchange rate publication. It requires no API key, has a stable JSON schema, and provides historical rates by date — which is why the previous day's rate can be fetched with a second call to the dated URL. Yahoo Finance's INR=X returns forex closing prices but wraps them inside the same equity chart JSON structure as stock tickers, making it more prone to returning market-session data that doesn't reflect the current interbank rate.

DevOps 9. test_imports.py — Startup Diagnostic Harness

A new utility file test_imports.py was created and committed as part of the Day 12(2) integration. Its purpose is to simulate the module-loading sequence that app.py performs at startup, reporting success or failure for each module individually.

The Problem It Solves

When app.py fails on startup (e.g., because a blueprint module has a syntax error, a missing file reference, or an incompatible import), the Gunicorn/Render error log shows a single unhandled exception at the create_app() call, with a stack trace that may be several levels deep. Identifying which of the nine editor modules caused the crash requires reading the trace carefully.

With test_imports.py, developers can run python test_imports.py locally or in an interactive Render shell to step through each module import and see exactly which one fails before deploying.

def _find_header_date(soup):
    hdr = soup.find(id="header")
    if not hdr:
        # Fallback: search by style if id is missing
        for div in soup.find_all("div"):
            style = div.get("style", "")
            if "color:#737373" in style and "font-weight:600" in style:
                if "Date:" in div.get_text():
                    return div
        return None
    return hdr.find("div", style=lambda s: s and "color:#737373" in s)

Design Note: Broadcast Architecture

The broadcast pattern — sending to all editors rather than a user-selected editor — is an intentional design choice. The user may decide after receiving stories which template layout to use for that day's edition. By pre-populating all editors simultaneously, the workflow becomes: extract → review stories → open whichever editor looks best → export. There is no separate "which editor?" routing step.

goodreturns.in Rate-of-Change Risk

The primary and fallback-1 tiers of the gold scraper both rely on parsing goodreturns.in's HTML table structure. If goodreturns.in changes its table layout (column ordering, number of columns, or the row identifier for 10g), both tiers will fail simultaneously and the COMEX fallback will activate. The COMEX rate is a global commodity price, not Mumbai retail — it may differ by 1–3% from the goodreturns.in value. The affected log line is DEBUG [Gold]: COMEX fallback -> INR <value>/10g.
