Developer Log Day 76–120

Recorded: June 26, 2026  |  Project: Pandora

Comprehensive documentation of the platform's transformation from a pure newsletter-editing toolkit into a full editorial & distribution suite. This log spans development days 76 through 120 — the period from commit 8798613 (where the Day 56–75 log closed) through the current working tree at commit 8b8610b. It covers the integration of Mailchimp analytics & remote campaign scheduling, the Social Pipeline orchestration dashboard, a second Git-native image pusher, live weather widgets, a Render → AWS Lightsail hosting migration, and a substantial dashboard re-design.

Executive Summary: Day 75 Condition vs. Day 120 Condition

Where We Were (Day 75 Condition)

At the close of Day 75, Project Pandora was a mature, SDK-free newsletter-editing platform. The editor fleet had grown to ten variants, the Gemini calls had been moved to direct REST, and the financial ticker had multi-tier fallbacks. But the platform stopped at the editor's Export HTML button — everything downstream of producing the newsletter was still manual:

  • No distribution path: Once a newsletter was exported, the editor copy-pasted the HTML into Mailchimp by hand to schedule or send it. The platform had no idea whether a campaign went out, to whom, or how it performed.
  • No analytics loop: Open rates, click rates, bounces, and unsubscribes lived entirely inside the Mailchimp web UI. There was no way to see campaign performance from within Pandora.
  • Unfinished work in the tree: The Day 9(2) and Day 17 editors were still unstaged at the end of the last log — fully built but uncommitted, with their app.py/dashboard.py/batch_extractor.py wiring pending.
  • Single image pipeline: Only the GitHub-REST upload-image pusher existed, serving images off raw githubusercontent.com URLs.
  • No weather: The dashboard card for Day 17 promised a "weather and markets ticker," but no live weather data source had been wired into any editor.
  • Hosting on Render only: The app ran exclusively on Render's ephemeral containers with no CI/CD; deploys were manual git push + Render auto-build.
  • Flat dashboard: A single undifferentiated grid of editor cards with one-line descriptions — no notion of which template to use on which day, and no editorial guidance.
Where We Are Now (Day 120 Condition)

Over the last ~45 development days the platform crossed the line from "makes the newsletter" to "ships the newsletter and tells you how it did." Every gap above has been addressed:

  • Mailchimp Campaign Analytics: A new /mailchimp blueprint pulls sent-campaign reports straight from the Mailchimp API — opens, clicks, bounces, unsubscribes and computed rates — with a 5-minute server-side cache and a 3-day look-back.
  • Remote Newsletter Scheduling: The new /schedule-mailchimp tool takes the exact ZIP/HTML you exported from any editor, lets you pick an audience, subject and sender, and either schedules it (IST → quarter-hour UTC) or sends it immediately — creating the campaign, pushing the HTML, and triggering the schedule/send action through three chained Mailchimp API calls. A live, scaled preview iframe renders the newsletter before dispatch.
  • Social Pipeline Dashboard: A runtime-state orchestration console at /social-pipeline with a file-backed state model (social_data/runtime/ vs social_data/persistent/), a single-source-of-truth reset, scheduler stop signalling, and a 15-second auto-polling status grid.
  • Git Image Pusher (v2): A second uploader (upload2.py, /git-pusher) that saves images locally and commits/pushes them with six selectable UI themes — complementing the original REST pusher.
  • Live Weather Widgets: Open-Meteo (geocoding + forecast) with a wttr.in JSON fallback, wired into the Day 15, Day 17, Day 12(2) and Day 9(2) editors, complete with WMO weather-code → icon mapping.
  • Hosting Migration → AWS Lightsail + Nginx: A GitHub Actions workflow now SSH-deploys to a Lightsail box (/var/www/flaskapp, systemd flaskapp service), and uploaded images are served from a self-hosted /uploads/ path that Nginx intercepts in production.
  • Day 9(2) & Day 17 finalized: Both editors are now committed and fully wired; the fleet is a stable, registered ten.
  • Export ZIP everywhere: Every editor gained an /api/export_zip endpoint, and the Code Viewer gained a "load directly from an editor" path via per-editor /api/preview routes.
  • Dashboard re-design: Editors are now grouped by Market Days vs Non-Market Days, each card carries a sentiment-coded editorial callout (.editor-note), and an AI persona ("Claude's two cents") prints one of six weekday-rotating quips per card.
  • First real documentation: README.md and PROJECT_CONTEXT.md were added, giving the project a proper module map, app map, and environment-variable reference.

Feature 2. Mailchimp Campaign Analytics

The first piece of the "what happens after Export?" story is read-only analytics. A new blueprint surfaces how recent campaigns performed without leaving Pandora. It is delivered in two files: mailchimp.py (a standalone, CLI-runnable data layer) and mailchimp_bp.py (the Flask blueprint with caching), backed by templates/mailchimp_dashboard.html.

API configuration via the key suffix

Mailchimp encodes the server/datacenter prefix as the suffix of the API key itself (e.g. ...-us21). Both modules derive the base URL from that suffix, so no separate datacenter config is needed:

def _api_config():
    api_key = os.environ.get("MAILCHIMP_API_KEY", "")
    if not api_key:
        raise RuntimeError("MAILCHIMP_API_KEY environment variable is not set.")
    server = api_key.split("-")[-1]                       # "us21"
    return (f"https://{server}.api.mailchimp.com/3.0",
            {"Authorization": f"Bearer {api_key}"})

Three-call report assembly

For a given date, _build_stats() pages through all sent campaigns in that day's window, fetches each campaign's report, and separately pages the unsubscribe list (which the summary report does not include as a running count). Per-campaign and aggregate rates are computed with a divide-by-zero-safe helper:

def _pct(part, total):
    return round(part / total * 100, 2) if total else 0.0

# per campaign: emails_sent, opens_total, clicks_total,
#               hard_bounces + soft_bounces, unsubscribe count
# → open_rate, click_rate, bounce_rate, unsub_rate

3-day look-back with a 5-minute cache

The /api/stats route is the smart part. If no explicit ?date= is supplied, it walks today → yesterday → day-before-yesterday and returns the first day that actually has campaigns — so the dashboard shows the most recent real send even on a quiet morning. Each resolved date is memoised in a module-level dict for 5 minutes to avoid hammering the Mailchimp API on every refresh:

_cache = {}
_CACHE_TTL = 300  # seconds

# for date in [today, yesterday, day_before]:
#     if cached & fresh: return cached
#     campaigns, totals = _build_stats(date)
#     if campaigns or explicit-date-requested:
#         cache and return
# else: return an explicit empty payload listing checked_dates
Error surfaces

A missing MAILCHIMP_API_KEY returns 500 with a clear message; any Mailchimp transport failure returns 502 with the upstream error text. The empty-state payload explicitly lists the checked_dates so the UI can say "no campaigns found for these three days" rather than rendering a blank table.

Feature 3. Schedule Mailchimp Newsletter — Remote Dispatch

This is the headline feature of the period and the most direct answer to the Day 75 gap. It closes the loop entirely: the file you export from any Pandora editor can now be uploaded, previewed, and scheduled straight into a Mailchimp audience — no manual copy-paste into the Mailchimp web UI. It lives at /schedule-mailchimp (schedule_mailchimp_bp.py).

Two states in this log

Commit 8b8610b ("Upcoming feature: Scheduling Newsletter Remotely") committed a 212-line placeholder — a styled "under construction" page with a diagonal hazard strip and a done/todo checklist. The full ~704-line implementation described below currently lives in the uncommitted working tree. It is functionally complete and is documented here as the real state of the feature.

Accepting the exported artifact (ZIP or HTML)

Editors export either a bare .html or a .zip (via the new Export ZIP, §7). _extract_html() transparently handles both, and when given a ZIP it prefers the shallowest, shortest-named HTML file and skips macOS resource-fork entries:

def _extract_html(file_storage):
    filename = (file_storage.filename or "").lower()
    data = file_storage.read()
    if filename.endswith(".zip"):
        with zipfile.ZipFile(io.BytesIO(data)) as zf:
            html_names = [n for n in zf.namelist()
                          if n.lower().endswith((".html", ".htm"))
                          and not n.startswith("__MACOSX")]
            if not html_names:
                raise ValueError("the ZIP does not contain an .html file.")
            html_names.sort(key=lambda n: (n.count("/"), len(n)))
            return zf.read(html_names[0]).decode("utf-8")
    return data.decode("utf-8")

IST → quarter-hour UTC conversion

The editor thinks in India Standard Time; Mailchimp requires schedule times in UTC, rounded to the quarter hour. _to_mailchimp_utc() handles the timezone shift and the 15-minute rounding, including the :60 roll-over edge case:

IST = timezone(timedelta(hours=5, minutes=30))

def _to_mailchimp_utc(local_str):
    naive = datetime.strptime(local_str, "%Y-%m-%dT%H:%M")
    utc = naive.replace(tzinfo=IST).astimezone(timezone.utc)
    rounded = int(round(utc.minute / 15.0) * 15)
    if rounded == 60:
        utc = utc.replace(minute=0, second=0, microsecond=0) + timedelta(hours=1)
    else:
        utc = utc.replace(minute=rounded, second=0, microsecond=0)
    return utc

The default send time is pre-filled to tomorrow at 9:00 AM IST, matching the team's standard morning dispatch.

The three-step dispatch

Mailchimp models a send as three separate API calls. /api/schedule chains them, surfacing a human-readable error at whichever step fails (and returning the created campaign_id so a half-built campaign can be found and cleaned up):

  1. Create campaign — POST /campaigns with type regular, the chosen list_id, subject, from-name and reply-to. The internal title is stamped with an IST timestamp for traceability.
  2. Push content — PUT /campaigns/{id}/content with the extracted html.
  3. Schedule or send — POST /campaigns/{id}/actions/schedule with the quarter-hour UTC time, or .../actions/send for immediate dispatch.

The schedule path validates that the requested time is at least a minute in the future before doing any work, and _mc_err() digs the detail/title/field-level errors out of Mailchimp's JSON error envelope so the operator sees "Send time must be in the future" rather than a raw 400.

Audience picker & live preview

/api/audiences lists every Mailchimp list with member counts and auto-selects the team's default audience (Push_160) by normalising names (stripping spaces/underscores, case-insensitive). /api/preview returns the extracted HTML so the front-end can render it inside a sandboxed iframe, auto-scaled to fit the panel width, with an "⤢ Expand" fullscreen overlay. The whole page is a single self-contained render_template_string with a newsprint aesthetic (Playfair Display + IBM Plex Mono, hazard-free this time).

Why this matters

Before this, "send the newsletter" was an off-platform manual ritual every single morning. Now it is: open the editor → Export ZIP → drop it on the scheduler → confirm the preview → pick 9 AM → done. The dashboard quip for this tool was updated from a self-deprecating "still in the workshop, sawdust everywhere" to "Quietly the most powerful button on this dashboard." It earned the promotion.

Architecture 4. Social Pipeline — Runtime-State Orchestration

The Social Pipeline is a runtime orchestration console for managing social-media post scheduling and publishing. It is split into a pure utility layer (social_utils.py) and a thin blueprint (social_pipeline_bp.py) serving a dashboard at /social-pipeline plus a status/reset API. The design principle is a clean split between transient and durable state.

State-file layout: runtime vs persistent

All pipeline data lives under social_data/, partitioned into two directories with very different lifecycles:

social_data/
├── runtime/          ← ALL files here are cleared on reset
│   ├── run_state.json          stage, in_progress, waiting_for_approval, current_post
│   ├── current_session.json    active session identity (id, started_at)
│   ├── scheduler_state.json    running, paused, next_run, stop_requested
│   ├── lock.json               orchestration mutex (owner, acquired_at)
│   ├── heartbeat.json          alive, last_beat
│   ├── queue_processing.json   processing, queue_size
│   ├── failed_run_marker.json  reason, failed_at
│   ├── last_send_result.json   last batch-extractor "Send to Template" result
│   └── *.lock                  any stray lock files
└── persistent/       ← NEVER touched by reset
    ├── approved_posts/   captions/   generated_images/
    ├── payload_archive/  archive/
    ├── posted_history.json
    └── duplicate_protection.json

Single source of truth for reset

reset_social_runtime_state() is the only code path that clears state. It deletes every file in the RUNTIME_FILES manifest plus any stray *.lock files, logs each deletion, and returns a {cleared, timestamp} receipt. Crucially it never touches persistent/ — posted history and duplicate-protection records survive a reset:

def reset_social_runtime_state():
    ensure_dirs()
    cleared = []
    for filename in RUNTIME_FILES:
        path = _runtime_path(filename)
        if os.path.exists(path):
            os.remove(path)
            cleared.append(filename)
            logger.info('[social-reset] cleared runtime state: %s', filename)
    for lock_file in glob.glob(os.path.join(RUNTIME_DIR, '*.lock')):
        os.remove(lock_file); cleared.append(os.path.basename(lock_file))
    logger.info('[social-reset] reset complete — cleared: %s', cleared)
    return {'cleared': cleared,
            'timestamp': datetime.datetime.utcnow().isoformat() + 'Z'}

Graceful scheduler stop, then reset

The reset endpoint is cooperative: if a scheduler is marked running, it first writes a stop_requested flag (and timestamp) into scheduler_state.json so the running loop can poll for it and exit cleanly, then wipes runtime state. get_pipeline_status() reads all eight runtime files and flattens them into one unified status dict consumed by the dashboard.

Dashboard behaviour

  • Auto-polls /api/social/status every 15 seconds; a Refresh button forces an immediate re-fetch.
  • Reset Pipeline is a red-outline button normally; when in_progress == true it becomes an amber Force Reset — never permanently disabled, so the operator can always break a stuck run.
  • A confirmation modal (with an extra amber warning when a run is in progress) guards the destructive action; success shows a green "Pipeline state reset successfully" banner.
Batch Extractor now feeds the pipeline

The integration point: batch_extractor.py's api_send() now writes its broadcast outcome to last_send_result.json in the runtime directory (timestamp, total, success count, per-endpoint results and failures). This means the "Send to Template" action is observable from the Social Pipeline status grid — see §11.

Feature 5. Git Image Pusher (upload2.py) — A Second Uploader

The original upload-image tool pushes images through the GitHub REST API and serves them off raw githubusercontent.com URLs. A second, complementary uploader was added at /git-pusher (upload2.py, git_pusher_bp) that takes the Git-native route: it saves the image locally into the app's uploaded_images/ directory and commits/pushes it to the same repository via subprocess git calls.

Local-first storage

REPO_PATH = os.path.join(os.path.dirname(os.path.abspath(__file__)),
                         "uploaded_images")
os.makedirs(REPO_PATH, exist_ok=True)

Because the image is written into the served directory first, it is available immediately at the self-hosted /uploads/<filename> path (§9) even before the push completes — the commit is for durability, not for serving.

Six selectable UI themes

The tool ships a themed front-end with six full colour palettes selectable at runtime — dark, light, dracula, forest, ocean and sunset — each defined as a block of CSS custom properties on a body.theme-* selector. It is the most visually configurable utility in the suite.

Two pushers, one purpose

Keeping both uploaders is deliberate: the REST pusher is simpler and survives ephemeral filesystems (it never relies on a local working tree), while the Git pusher keeps the repository as the single source of truth for assets and pairs naturally with the new self-hosted /uploads/ serving on Lightsail. The dashboard quips lean into the rivalry — the Git pusher is framed as "Image Uploader's overachieving cousin who insists on doing it 'the proper way' with version control."

Feature 6. Live Weather Widgets (Open-Meteo + wttr.in)

The Day 17 dashboard card had long promised "weather and markets ticker," but no weather data source existed. This period wired a real, key-free weather pipeline into four editors: Day 15, Day 17, Day 12(2) and Day 9(2). The Day 12(2) weather section landed last, in commit ec767fc (June 9).

Two-tier, no-API-key sourcing

Each editor exposes a weather-fetch route that resolves a place name to a forecast through a primary/fallback chain — neither tier requires an API key, keeping the dependency surface at zero:

  1. Tier 1 — Open-Meteo. Geocode the location via geocoding-api.open-meteo.com/v1/search, then request daily=temperature_2m_max,temperature_2m_min,weather_code from api.open-meteo.com/v1/forecast with timezone=auto&forecast_days=2.
  2. Tier 2 — wttr.in. If Open-Meteo fails, fall back to wttr.in/<loc>?format=j1, reading maxtempC/mintempC and the mid-day hourly weatherDesc/weatherCode.

Resilience details worth noting

# Open-Meteo renamed weathercode → weather_code; support both
codes = daily.get("weather_code") or daily.get("weathercode", [])
...
# WMO code → (description, icon)
desc, icon = _WMO_CODES.get(wmo, ("Partly cloudy", "⛅"))
  • Schema drift guard: Open-Meteo renamed weathercode to weather_code; the code accepts either key so an upstream rename can't blank the widget.
  • WMO icon map: a _WMO_CODES table maps numeric WMO weather codes to a human description and an emoji icon.
  • Mumbai temperature sanity clamp: a clean_temp() helper coerces values to integers and nudges implausibly low Mumbai readings up to a sane floor — a guard against a bad upstream reading printing "12°C" in a tropical-city newsletter.
  • Never blank: if both tiers fail, the route still returns sensible defaults with a warning field and HTTP 200, so the editor renders a usable widget rather than an error.

Feature 7. Export ZIP Everywhere & "Load From Editor"

def api_export_zip():

The most distinctive addition. A CLAUDE_QUOTES dict holds six quips per card — one for each weekday Mon→Sat — for every editor and utility tool. The dashboard route picks the slot matching today's weekday so the copy stays fresh daily (Sunday falls back to Saturday's):

@dashboard_bp.route('/dashboard')
@require_login
def dashboard():
    idx = min(datetime.now().weekday(), 5)        # Mon=0 … Sat=5
    todays_quotes = {k: v[idx] for k, v in CLAUDE_QUOTES.items()}
    return render_template_string(DASHBOARD_HTML,
                                  logo_url=config.LOGO_URL, q=todays_quotes)

The quips are an editorial personality layer — the Day 15 template is "the valedictorian … peaked in high school and somehow keeps peaking," Day 11 gets "pour one out, it's not dead, it's resting (it's dead)," and the News Extractor is "the manual transmission in an automatic world." Migration 9. Hosting Migration: Render → AWS Lightsail + Nginx

A quieter but architecturally significant shift: the platform gained a real CI/CD path to a self-managed server, moving off Render's build-on-push model toward an Nginx-fronted systemd service on AWS Lightsail. (The README.md / PROJECT_CONTEXT.md still describe Render as the host — documentation lags the infra here, noted under Known Issues.)

GitHub Actions SSH deploy

Commit 0157f69 added .github/workflows/deploy.yml — on every push to main, it SSHes into the server, pulls, reinstalls deps, and restarts the service via systemd:

name: Deploy to Lightsail
on:
  push: { branches: [ main ] }
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ubuntu
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            cd /var/www/flaskapp
            git pull origin main
            source venv/bin/activate
            pip install -r requirements.txt
            sudo systemctl restart flaskapp

Self-hosted image serving via /uploads/

To match the new host, app.py added a Flask route that serves the uploaded_images/ directory at /uploads/<filename>, with the explicit understanding that Nginx intercepts this path in production and Flask only handles it in dev/fallback:

# Serve uploaded images — Nginx intercepts this in production;
# Flask handles it in dev/fallback.
_upload_dir = os.path.join(os.path.dirname(os.path.abspath(__file__)),
                           "uploaded_images")

@app.route('/uploads/<filename>')
def serve_upload(filename):
    return send_from_directory(_upload_dir, filename)

Correspondingly, upload_image.py was refactored (commit d0de69d) so the URL it hands back points at the self-hosted {base_url}/uploads/ path instead of a raw GitHub URL, the success copy changed from "saved at" to "available at," and a cleanup_old_images() helper was added to keep the directory from growing unbounded on a persistent disk.

Why move off raw GitHub URLs

Serving newsletter images from githubusercontent.com works but couples asset delivery to GitHub's CDN and rate limits, and makes the URLs awkward for special-character filenames. Serving from the app's own /uploads/ path (Nginx in front) gives stable, predictable URLs on the same origin as the editors — better for email clients and for the preview iframes.

Bugfix 10. Stability Fixes

Python 3.11 f-string regex crash (commit d0de69d)

Backslash inside an f-string expression

Several modules shared a fix_date_br() helper that embedded a regex substitution directly inside an f-string expression. On Python < 3.12 a backslash is illegal inside the {...} portion of an f-string, so the file raised a SyntaxError at import time on the 3.11 runtime — taking down any blueprint that imported it.

# Before — SyntaxError on Python 3.11 (backslash in f-string expr)
text = f"{day}<br>{re.sub(r'^\s*<br\s*/?>\s*', '', rest, flags=re.I)}"

# After — extract to a variable first
cleaned_rest = re.sub(r'^\s*<br\s*/?>\s*', '', rest, flags=re.I)
text = f"{day}<br>{cleaned_rest}"

The fix was applied consistently across backup.py, editor(for Day8.html).py, helpers.py and the other files carrying the shared helper.

Day 15 news-card glitch & batch-extractor glitch (commit a1b3f90)

The Day 15 editor could throw when a target element it expected to update was missing from the parsed DOM (a "missing target element" condition) — the save path now guards for the absent node instead of dereferencing None. The same commit cleared a glitch in the batch extractor's flow.

Market data: pct float over pre-formatted strings (commit 843e47b)

The markets layer was refactored to pass the change percentage around as a raw float rather than a pre-formatted display string. Carrying the numeric value means the sign (and therefore the up/down colour) is decided at render time from the number itself, rather than re-parsing a string like "+1.2%" — removing a class of formatting/colour-mismatch bugs. The same commit corrected Day 17's S2–S5 link and paragraph extraction.

Refactor 11. Batch Extractor: Broadcast Result Telemetry

The api_send() broadcast in batch_extractor.py was upgraded in two ways. First, its forwarding fleet was extended to all ten editors (adding the now-committed Day 9(2) and Day 17 endpoints). Second — and more importantly — it now persists a structured record of every broadcast into the Social Pipeline's runtime state:

from social_utils import write_runtime_file

results = []
# ... POST payload to each of the 10 /api/import_json endpoints ...
#     results.append({"endpoint": target, "ok": True/False, "reason": ...})

failed_details = [r for r in results if not r['ok']]
write_runtime_file('last_send_result.json', {
    'timestamp': datetime.datetime.utcnow().isoformat() + 'Z',
    'total': len(endpoints),
    'success_count': success_count,
    'results': results,
    'failed': failed_details,
})

The old design only tracked a flat errors list. The new design records a per-endpoint {endpoint, ok, reason} object for all ten targets, computes the failed subset, and writes the whole thing to last_send_result.json. get_pipeline_status() surfaces it as send_total / send_success_count / send_results / send_failed — so the Social Pipeline dashboard can show whether the morning's "Send to Template" actually reached every editor. This is the concrete wiring that connects the extractor to the new orchestration layer (§4).

Known Issues & Open Items at Day 120

Mailchimp scheduler is uncommitted

The full ~704-line implementation of schedule_mailchimp_bp.py (audiences, live preview, IST→UTC scheduling, send-now) and the matching dashboard copy flip are in the working tree only. Commit 8b8610b shipped a placeholder. Stage and commit both before the next Lightsail deploy, or the deployed app will still show "under construction."

Docs say "Render," infra says "Lightsail"

README.md and PROJECT_CONTEXT.md both list Render as the host, but deploy.yml targets a Lightsail box with Nginx and a systemd flaskapp service, and image serving assumes Nginx intercepts /uploads/. The documentation should be reconciled with the actual deploy target, and the dual-host story (is Render still a live fallback, or fully retired?) clarified.

scratch/ is in the repo

~17 one-off recovery/inspection scripts plus a recovered_Day17.html snapshot are tracked under scratch/. They are dev-only and should be moved behind .gitignore (or deleted) so they stop appearing in diffs and file counts.

social_data/persistent durability on ephemeral hosts

As PROJECT_CONTEXT.md notes, social_data/persistent/ (posted history, duplicate protection, approved posts) needs a mounted disk or external storage to survive a container restart. On the new Lightsail box this is a real local disk; if Render remains a fallback, persistence there is not guaranteed.

No campaign de-duplication in the scheduler

The scheduler creates a fresh Mailchimp campaign on every submit. Double-submitting the same exported file (or a flaky network retry after step 1 succeeds) can leave orphaned half-built campaigns in Mailchimp. The response returns the campaign_id for manual cleanup, but an idempotency guard or a "you already scheduled this today" check would be safer.

Unprotected routes — still open from Day 75

Auth is a single session login; the carried-over recommendation to add explicit route guards still applies, now across a wider surface (four new blueprints, including the destructive Social Pipeline reset and the live Mailchimp dispatch).

Conclusion & Next Steps

Days 76–120 were the period Project Pandora outgrew the word "editor." The platform that began the window as a sophisticated HTML-newsletter builder ended it as an end-to-end editorial pipeline: extract the stories, build the newsletter in one of ten templates, export it, schedule it into Mailchimp, watch the open and click rates come back, and orchestrate the social side — all without leaving the dashboard. Along the way it picked up a real CI/CD path to a self-managed Lightsail host, live weather, a second image pipeline, its first documentation, and a dashboard with genuine editorial opinions.

The architecture proved it can absorb whole new product surfaces — analytics, scheduling, orchestration — as cleanly registered blueprints, not bolt-ons. The investment focus for Days 121+ should shift from "add capability" to "harden and finish."
